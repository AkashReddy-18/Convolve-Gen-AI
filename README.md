# Invoice Information Extraction using Qwen2-VL

An end-to-end Visual Language Model (VLM) pipeline for extracting structured information from invoice images and PDF documents using the **Qwen/Qwen2-VL-2B-Instruct** model.

## Project Overview
This project automates the extraction of key financial and metadata fields from invoices. It is designed to handle complex document features including:
*   **Multilingual Support:** Recognition of English and Indian languages (Hindi, Tamil, etc.).
*   **Handwritten Elements:** Detection of handwritten signatures.
*   **Official Markers:** Detection and localization of stamps and seals.
*   **Multimodal Input:** Native processing of both PDF and image formats.

## Technical Specification

### Model Details
*   **Model Name:** `Qwen/Qwen2-VL-2B-Instruct`
*   **Architecture:** Vision-Language Transformer
*   **Inference Mode:** Conditional generation with JSON-targeted prompting.
*   **Environment:** Optimized for GPU-accelerated environments (Google Colab T4/A100).

### Extracted Fields
The pipeline extracts the following schema for every document:
```json
{
  "dealer_name": "string or null",
  "model_name": "string or null",
  "horse_power": "number or null",
  "asset_cost": "number or null",
  "stamp": {
    "present": "boolean",
    "bbox": [x1, y1, x2, y2]
  },
  "signature": {
    "present": "boolean",
    "bbox": [x1, y1, x2, y2]
  }
}
```

### Critical Business Rules
*   **Stamp→Signature Rule:** If a stamp is detected but a signature is missing, the stamp's bounding box is programmatically reused for the signature field to ensure document validity.
*   **Confidence Scoring:** A rule-based heuristic (0.0 to 1.0):
    *   Text/Numeric fields (Dealer, Model, HP, Cost): +0.2 each.
    *   Visual markers (Stamp, Signature): +0.1 each.

## Repository Structure
The repository is organized into a single `Final_submission/` directory for simplicity:
*   `Final_submission/main.py`: Main development and execution notebook. Contains the complete model pipeline, preprocessing logic, and unit tests.
*   `Final_submission/requirements.txt`: Python dependencies required to run the notebook.
*   `Final_submission/result.json`: The consolidated output of the extraction from the `train/` dataset.
*   `Final_submission/README.md`: This documentation.

## Execution Guide

### Option 1: Google Colab (Recommended)
1. Open `main.py` in Google Colab.
2. Change the Runtime type to **GPU** (T4 or higher).
3. Run all cells. The notebook will automatically install dependencies from the code cells, load the model, and process your invoices.
4. Results will be saved and can be downloaded as JSON.

### Option 2: Local Jupyter Environment
Ensure you have a GPU-enabled Linux environment with Python 3.10+.
1. **Install Dependencies:**
   ```bash
   pip install -r Final_submission/requirements.txt
   sudo apt-get install poppler-utils  # Required for PDF processing
   ```
2. **Open the Notebook:**
   ```bash
   jupyter notebook Final_submission/main.py
   ```

## Limitations & Notes
*   **Approximate Bounding Boxes:** Bounding boxes are generated by the 2B model and are approximate; they provide region hints rather than pixel-perfect precision.
*   **Heuristic Confidence:** The confidence score represents field presence, not probabilistic certainty.
*   **Hardware:** 12GB+ VRAM is recommended for stable performance.

## Recent Improvements & Audit Fixes
The current version has been audited for strict compliance:
1.  **Schema Enforcement:** Fixed the output to produce a flat list of 5-key objects as per specification.
2.  **Robust Confidence:** Corrected logic to properly score numeric `0` values as present.
3.  **Bbox Integrity:** Programmatically enforced the Stamp→Signature bbox reuse rule to prevent model hallucinations.
4.  **Full Filenames:** Updated document IDs to include file extensions (e.g., `invoice.pdf`) for better traceability.

### Fallback Strategy
If Qwen2-VL consistently fails for a specific document type, the following fallback is recommended:
1. **Secondary:** Tesseract OCR (with `tesseract-ocr-hin` for Hindi support) for text-only extraction.
2. **Tertiary:** PaddleOCR for improved multilingual/tabular recognition.
3. **Marker Detection:** Use a dedicated YOLOv8 model for high-precision stamp/signature localization if bboxes require pixel-perfect accuracy.
